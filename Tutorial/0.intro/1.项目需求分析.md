
## 项目需求分析

### 一、业务场景分析

#### 1 业务目标

利用给到的数据中的品牌、类目（如连衣裙、儿童衬衫、羽绒服）、属性、长文案、价格等结构化+非结构化信息，构建服饰为主的商品知识库，提升搜索与导购转化率、减少人工客服。

#### 2 用户交互
用户通过搜索框或聊天自然语言提问（如“灰色春季通勤西装裤套装 1500 左右”“3-6 岁夏季纯棉儿童衬衫”），系统给出可解释的商品推荐与对比；运营同样可查询某品牌/价位/品类分布。

#### 3 评估与指标

从召回与匹配准确率、问答正确率（规格/材质/季节回答是否正确）、点击率/转化率以及平均响应时间等维度衡量效果。

### 二、应用类型选择

#### 1 对话系统（ChatBot）
基于商品知识库的智能导购与客服，围绕尺码、材质、季节、适用场景、洗涤方式、价格区间等进行多轮问答和推荐。

#### 2 内容生成（Content Gen）

利用 product_description 与 marketing_attributes 自动生成多版本标题、卖点 bullets、搜索关键词和渠道差异化文案，保持品牌/类目风格一致。
#### 3 知识问答 + 文档理解（QA / Doc Analysis）

对长文案做属性抽取和归一（面料成分、风格、场景、弹性、厚度等），支持“按材质/季节/风格/价格”组合检索，并生成品类与价格带分析报告。

### 三、技术约束评估

#### 1 计算资源与延迟

- 商品量级较大，需要支持结构化过滤 + 向量检索的索引（可结合 OpenSearch）；
- 面向 C 端对话与搜索目标 P95 延迟控制在 1–2 秒内，运营分析可以离线批处理。


#### 2 数据安全与成本

- 商品数据本身是公开信息，但与用户行为/订单数据结合时需做脱敏、权限控制；
- 高频检索与简单问答优先用本地向量库+小模型，长文案生成或复杂推理再调用大模型以控制 token 与调用成本。


#### 3 可扩展性

在现有 skuid/spuid/brand_name/group_name/price 等 schema 上预留图片 URL、库存、评分等字段，索引与服务架构需支持后续多品类扩展与跨语种文案。

### 四、场景定制化

#### 1 领域知识整合

将服饰类目树、尺码对照表、季节与场景（商务通勤、聚会、校园、运动等）词库、价格带策略等显式建模，并与 LLM 检索与推理结合，减少误匹配。

#### 2 多模态需求分析
当前以文本（标题、营销属性、长描述）为主做“以文搜衣”，后续可引入商品图片实现图文联合检索与“看图找同款”“找相似风格”等能力。


#### 3 个性化定制
结合用户画像和历史行为，对 group_name、price、品牌偏好等做个性化排序与讲解，例如为“学生预算”“商务白领”“高端轻奢”生成不同推荐理由与话术。
